# Stakeholder Analysis

Use this reference when the CTMF is stakeholder analysis or when a draft is too generic and needs a stronger project-grounded chain.

## Core Test

The entry should show:
- which stakeholder group mattered
- what concern they raised
- what NGO / requirement / EC came from it
- where that requirement appeared later
- what concept became weaker or stronger because it stayed visible
- what stakeholder analysis still could not reveal

If any of those are vague, the section is not grounded enough.

## Strong Chain

Use this chain explicitly:

stakeholder -> concern -> NGO / requirement -> design consequence -> later concept decision

Without that chain, the section reads as "we did stakeholder analysis because design teams do stakeholder analysis."

## Praxis I Pattern

The strongest Praxis I stakeholder-analysis move was not "students confirmed the can is loud."

It was:
- primary student conversations surfaced additional concerns the team had not foregrounded strongly enough
- those concerns included spill control, beverage disturbance, awkward interaction, and operational burden
- those concerns then became explicit goals, requirements, and evaluation criteria
- later concept comparison changed because the project was no longer judged on noise alone

That is the difference between decorative stakeholder work and real engineering consequence.

## Example Evidence Logic From Praxis I

### Beverage integrity / spill control

Look for evidence like:
- interview comments about the can "bursting out"
- worries about contents getting on the user's hands
- worries about the can spilling all over the place

These should map to later artifacts such as:
- beverage-integrity goals
- liquid-loss requirements
- CO2-loss requirements
- evaluation criteria on spill / boil-off control

### Ease of use / awkward interaction

Look for evidence like:
- comments about awkward finger interaction
- "digging it under the top"
- opening taking too long

These should map to later artifacts such as:
- opening-time requirements
- gesture limits
- force limits
- hand-pressure limits
- accessibility or usability ECs

## Strong Interpretive Claims

Prefer claims like:
- "Stakeholder analysis expanded the problem from a narrow acoustic challenge into a broader user-centered engineering problem."
- "Primary conversations surfaced spill control and interaction quality as requirements the team would not have foregrounded as strongly otherwise."
- "Stakeholder analysis clarified what success had to protect, but it could not reveal the physical sound-generation mechanism."

Avoid claims like:
- "Stakeholder analysis was useful."
- "It helped ground the project in people."
- "It informed the process."

Those are too general unless they are immediately tied to a project consequence.

## Best Process Fit

For stakeholder analysis, the strongest stage is usually:
- `Frame` first
- `Represent` second

Use `Converge` only when you can show the later comparison or recommendation changed because the stakeholder-derived requirements stayed active.

## Distinguish What It Did vs What It Could Not Do

Use both.

Good pattern:
- stakeholder analysis told the team what success had to protect
- it did not tell the team the physical mechanism causing the problem
- therefore it had to be paired with testing or another CTMF

That is what makes the assessment legitimate instead of celebratory.

## Prompt Set For Eliciting Missing Evidence

Ask for short, exact answers to these:

1. Which stakeholder group mattered most?
2. What exact concern did they raise?
3. What exact NGO, requirement, objective, or EC came from that concern?
4. Where did that requirement show up later: framing, requirements, concept screening, Pugh chart, testing, or final recommendation?
5. Which concept would have looked better if that concern had been ignored?
6. Which concept became stronger because it stayed visible?
7. What artifact proves this chain happened?
8. What did stakeholder analysis help the user see clearly?
9. What did stakeholder analysis fail to reveal?
10. What tendency of the user's design approach did this CTMF support?
11. What bias did it guard against?
12. What would the user do differently next time?

## Outline For A Strong Entry

Use this sequence:

1. CTMF snapshot
2. Core claim
3. What stakeholder analysis is
4. How it was used in the project
5. Stakeholder concern -> requirement mapping
6. Where it appeared later in the process
7. What changed in concept selection
8. What it clarified
9. What it failed to reveal
10. Fit with position and bias
11. Future-use rules
12. Evidence / visual stack

## Final Quality Rule

Every paragraph in the entry should point to a specific project activity, artifact, decision, or failure mode. If it could apply equally well to any generic stakeholder-analysis exercise, it is still too weak.
