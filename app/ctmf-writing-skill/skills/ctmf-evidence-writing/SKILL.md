Use this skill to convert a weak CTMF section from "I understand this tool" into "this tool did specific work in this project."

## Goal

Make the CTMF entry prove four things:
- why this CTMF belongs in this project specifically
- what design activity or decision it changed
- what it clarified and what it failed to reveal
- how it supports the user's strengths and guards against a bias

## Workflow

1. Load project evidence before drafting.
- Read the current CTMF draft.
- Read rubric language or assessment criteria if available.
- Pull project artifacts that show actual use: report sections, requirement tables, interview transcripts, concept matrices, calculations, testing outputs, review comments, and relevant visuals.

2. Diagnose the weakness.
- Mark any sentence that could apply to any generic CTMF exercise.
- Mark where the draft names the tool but does not show a project consequence.
- Mark where the draft explains structure instead of making an argument.

3. Build the project chain.
- For any CTMF, force this sequence:
  project activity or evidence -> CTMF move -> requirement / criterion / design consequence -> later concept or decision effect
- For stakeholder analysis specifically, use:
  stakeholder -> concern -> NGO / requirement -> design consequence -> later concept decision

4. Separate three different claims.
- What the CTMF helped define.
- What the CTMF changed later in the process.
- What the CTMF could not reveal on its own.

5. Connect to FDCR honestly.
- `Frame`: the CTMF changed what the problem was.
- `Represent`: the CTMF turned concerns into goals, requirements, ECs, or other explicit artifacts.
- `Diverge`: the CTMF widened or redirected concept generation.
- `Converge`: the CTMF changed concept screening, comparison, or recommendation.

6. Connect to the user's position.
- State which design tendency the CTMF supports.
- State which bias it guards against.
- Make the bias concrete, not abstract.

## Grounding Rules

- Every paragraph should point to a specific project activity, artifact, decision, or failure mode.
- If a sentence could fit any stakeholder analysis, Pugh chart, requirement matrix, or generic design report, it is still too superficial.
- Do not overclaim solo authorship when the work was collaborative. Describe the user's role as part of shared team process when that is the truth.
- Prefer "this surfaced", "this translated into", "this later changed" over "this was useful" or "this helped generally."
- Prefer failure modes and downstream consequences over values language.

## Drafting Order

Use this sequence unless the user explicitly asks for another format:

1. CTMF snapshot
2. Core claim
3. What the CTMF is
4. How it was used in the project
5. Evidence chain
6. Where it appeared later in FDCR / process
7. What changed because it stayed visible
8. What it clarified
9. What it failed to reveal
10. Fit with position and bias
11. Future-use rules
12. Evidence / visual stack

## Review Questions

Before finalizing, check whether a reader can answer all of these without guessing:
- Which project artifact proves the CTMF was used?
- What exact requirement, criterion, or concept decision came out of it?
- Where in the process did it matter most?
- What later decision did it influence?
- What could it not reveal?
- Why does it fit the user's engineering position?
- What bias did it guard against?

## When To Read References

- For stakeholder-analysis CTMF work, read [references/stakeholder-analysis.md](references/stakeholder-analysis.md).
- If the user provides a weak CTMF draft plus project context, use the reference questions as an elicitation tool before rewriting.
